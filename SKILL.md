# UX Evaluator — 网页体验测评

## 概述

本技能对公有云/SaaS 网页进行系统化的用户体验测评，输出一份专业的 HTML 格式测评报告。测评面向 **SMB 客户**视角，重点关注**易用性**和**购买转化率**。

## 使用场景

- 用户提供了一个网页 URL，希望评估其体验质量
- 用户上传了页面截图/设计稿，希望获得体验评审
- 用户正在设计一个公有云购买页面，需要转化率优化建议
- 用户想对比竞品页面的体验差异
- 用户询问"这个页面有什么体验问题？"

## 测评流程

### 第 0 步：识别页面类型（必须）

**在开始测评前，必须先判断页面类型**，然后启用对应的评估策略和权重体系。详见 `references/evaluation-framework.md`「页面类型识别」章节。

| 类型 | 关键特征 | 权重侧重 |
|------|---------|---------|
| **品牌宣传类** | Hero大图、品牌故事、客户案例 | 视觉35% + 文案35% |
| **购买类** | 价格表、CTA、套餐对比 | 文案35% + 漏斗30% |
| **操作类** | 表单密集、数据表格、控制台 | 交互40% + 视觉30% |
| **资料类** | 文档层次、代码块、侧目录 | 内容可读性40% + 操作完整性35% + 内容精简度25% |

识别后，在报告 Masthead 中通过 `{{PAGE_TYPE}}` 和 `{{TYPE_BADGE}}` 标注页面类型。后续所有评分必须使用该类型对应的权重计算。

### 第 1 步：获取页面信息

根据用户提供的内容获取页面：

- **URL**：使用 WebFetch 工具获取页面内容。重点关注可见文本、按钮文案、导航结构、定价信息、CTA 位置等
- **截图/图片**：使用 Read 工具查看视觉设计。关注色彩、排版、视觉层级、留白、品牌一致性
- **HTML 代码**：如果用户提供了 HTML 文件，直接读取分析

### 第 2 步：执行系统化测评

根据第 0 步确定的页面类型，使用对应的权重体系进行四维度测评。各维度子维度清单见 `references/evaluation-framework.md`。

**NA 评分规则**：部分子维度在某些页面类型下不适用。参考 `references/evaluation-framework.md`「维度适用性矩阵」：
- 操作类控制台的 F2（定价清晰度）和 F5（紧迫感）可标记 NA
- 无表单页面 I4（表单体验）可标记 NA
- 每页最多 2 个 NA，不得将低分项标记为 NA 抬高总分
- NA 维度不计入维度均分，在 methodology 表格中以灰色 N/A 标注

测评时必须引用具体的 UX 原则作为依据（详见 `references/principles.md`）。每个问题的分析必须指明「观察到的现象 → 违反/符合的原则 → 对 SMB 客户的具体影响 → 改进方向」。

**SMB 场景特别关注**：测评时主动检查以下高发问题（详见 `references/principles.md` 第七章）：
- 🚨 恐惧式文案 — 风险警告只有后果无缓解措施
- 🚨 账单恐惧 — 操作页面缺费用信息导致不敢操作
- 🚨 小触摸目标集群 — 密集小按钮在笔记本上极易误触
- 🚨 双重视觉语言 — 多套设计系统同屏产生割裂感
- 🚨 工具提示缺失 — 技术选项无内联解释导致决策瘫痪

**四维度权重由页面类型决定**（详见 `references/evaluation-framework.md`）：

| 维度 | 品牌宣传类 | 购买类 | 操作类 | 资料类 |
|------|----------|--------|--------|--------|
| 🎨 视觉层 | 35% | 15% | 30% | — |
| 🖱 交互层 | 20% | 20% | 40% | — |
| ✍️ 文案层 | 35% | 35% | 20% | — |
| 📈 转化漏斗 | 10% | 30% | 10% | — |
| 📖 内容可读性 | — | — | — | 40% |
| 📝 内容精简度 | — | — | — | 25% |
| 🔧 操作完整性 | — | — | — | 35% |

#### 维度一：视觉层（权重 20%）

| 子维度 | 检查要点 |
|--------|---------|
| 品牌一致性 | 色彩、字体、图标风格是否统一？是否符合品牌调性？ |
| 色彩系统 | 主色/辅色/中性色体系是否清晰？对比度是否满足 WCAG AA（4.5:1）？ |
| 排版层级 | 标题/副标题/正文层级是否分明？字号梯度是否合理？行高、段间距是否舒适？ |
| 视觉重心 | 首屏核心信息是否在 3 秒内可识别？视觉流是否引导用户到达 CTA？ |
| 图标与插画 | 风格是否统一？是否有语义辅助作用还是纯装饰？ |

视觉层引用原则：格式塔接近性/相似性/连续性、About Face 4 视觉层级、60-30-10 配色法则。

#### 维度二：交互层（权重 30%）

| 子维度 | 检查要点 |
|--------|---------|
| 导航清晰度 | 主导航是否在 3 步内到达任意页面？当前位置是否有明确指示（面包屑/高亮）？ |
| 费茨定律 | 关键 CTA 按钮尺寸是否≥44×44px（移动端）或≥32×32px（桌面端）？高频操作是否靠近用户光标自然位置？ |
| 希克定律 | 选择项是否过多（如定价卡片>4 个）？是否有默认推荐减少决策负担？ |
| 表单体验 | 输入框是否有明确标签？是否有实时校验反馈？是否支持 Tab 键顺序？ |
| 反馈及时性 | 点击后是否有即时视觉反馈（hover/active 态）？加载状态是否明确（骨架屏/进度条）？ |
| 错误与空状态 | 错误提示是否指明原因+解决方式？空状态是否有引导操作？ |

交互层引用原则：费茨定律、希克定律、尼尔森可用性原则（系统状态可见性、用户控制与自由、错误预防、识别而非回忆）。

#### 维度三：文案层（权重 25%）

| 子维度 | 检查要点 |
|--------|---------|
| 标题吸引力 | 主标题是否在 5 秒内传达核心价值主张？是否使用 SMB 客户的语言而非技术术语？ |
| 卖点表达 | 是否将功能转化为利益？（"自动备份"→"数据丢了也能一键恢复"） |
| CTA 有效性 | 按钮文案是否使用动词+利益？（"免费试用"优于"提交"）CTA 周围是否消除了用户疑虑？ |
| 语气一致性 | 全文语气是否统一？（专业但不生硬，友好但不轻浮） |
| 信任与安全感 | 是否有消除疑虑的文案？（"无需信用卡""7 天无理由退款""SOC2 认证"） |

文案层引用原则：微文案设计原则、About Face 4 用户心智模型匹配、CTA 转化文案最佳实践。

#### 维度四：转化漏斗（权重 25%）

| 子维度 | 检查要点 |
|--------|---------|
| 购买路径 | 从首页到完成购买的点击次数是否≤5 步？路径中是否有不必要的干扰？ |
| 定价清晰度 | 价格是否在首屏可见？不同套餐差异是否一目了然？是否有 FAQ 解答定价疑虑？ |
| 信任背书 | 是否有客户 logo 墙/案例/数据证明？是否有权威认证标识？ |
| 决策成本 | 是否提供免费试用/演示/快速体验入口？是否有退款保证降低尝试门槛？ |
| 紧迫感 | 是否有合理的紧迫感设计（限时优惠）而不显 manipulative？ |

转化漏斗引用原则：Fogg 行为模型（B=MAT）、Cialdini 说服原则（社会认同、权威、稀缺性）。

### 第 3 步：生成 HTML 报告

**生成前预检清单（必须逐项确认）**：

| # | 检查项 | 验证方法 |
|---|--------|---------|
| 1 | 页面类型判定正确 | 对照「三看」流程复核 |
| 2 | 权重公式匹配页面类型 | 核对 `evaluation-framework.md` 对应章节 |
| 3 | NA 维度 ≤2 个且有充分理由 | 每个 NA 附一句话说明 |
| 4 | 每项评分均有 UX 原则引用 | 搜索报告中是否存在无 `📐 依据` 的 dim-item |
| 5 | 改进建议含 P0/P1/P2 标记 | P0 仅用于阻断性问题（数据丢失/安全风险/核心流程不可完成） |
| 6 | 侧边栏 nav id 与 section id 一一对应 | 检查 `#score-overview` `#visual` `#interaction` `#copy` `#funnel` `#strengths` `#improvements` `#summary` `#checklist` |
| 7 | 所有 `{{占位符}}` 已替换 | 搜索 HTML 源码中是否残留 `{{` |
| 8 | 综合评分与仪表盘颜色匹配 | ≥80 绿 #1a7f5a · 60-79 琥珀 #c8892b · <60 红 #c0453a |
| 9 | 清单项与页面类型匹配 | 操作类清单 ≠ 购买类清单 |
| 10 | checklist 每项有 ✓/✗/△ 状态标记 | 搜索 `.status.pass` `.status.fail` `.status.warn` |

> 预检未通过的项目，在报告生成前修复。预检通过后再输出最终 HTML。

读取 `assets/report-template.html` 模板，将测评结果填入。

模板采用 **「编辑权威」设计风格**（Editorial Authority）——深色墨蓝基底 + 琥珀色强调 + Playfair Display 标题字体 + SVG 环形仪表盘，呈现顶级咨询公司报告质感。包含噪声纹理叠加、渐变装饰线、微交互 hover 态等精致细节。

**左侧固定导航栏**：报告左侧有 sticky 定位的章节目录，读者可一键跳转到任意章节。支持：
- Scroll-spy：滚动时自动高亮当前章节
- 移动端：底部浮动按钮 ☰ 唤起侧边抽屉式导航，点击遮罩关闭
- 所有内容章节已设置 `id` 锚点，导航链接使用 `href="#id"` 跳转

## 评分标准

整体采用百分制。每个维度的子维度平均后得到维度分，再按权重汇总。

**分数区间含义：**
- 90-100：行业标杆，SMB 客户可以无障碍完成购买
- 80-89：体验优秀，少数细节可优化
- 70-79：尚可使用，但存在明显的转化障碍
- 60-69：体验较差，SMB 客户很可能中途放弃
- <60：存在严重的可用性问题，需立即整改

## 重要原则

1. **必须提供原则依据**：每个结论都引用至少一条 UX 原则或定律，不要凭感觉评价
2. **面向 SMB 客户**：评价角度始终是「一个非技术背景的中小企业主能否顺利使用」
3. **量化优先**：能用数字就不用形容词（"按钮只有 28px" 优于 "按钮太小"）
4. **可执行性**：改进建议必须具体到「改什么→改成什么→为什么」，而不是泛泛的「优化体验」
5. **报告即交付物**：HTML 报告可以直接发给产品团队作为需求文档，无需二次加工
6. **NA 慎用**：不适用维度最多 2 个，需附理由。不得通过标记 NA 来抬高总分
