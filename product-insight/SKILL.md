# 产品洞察与战略规划

## 任务目标
- 本 Skill 用于：系统化分析产品的战略定位、目标客群、商业模式、关键竞争力和用户体验
- 能力包含：五维深度洞察、结构化分析框架、战略规划输出
- 触发条件：需要做产品战略规划、竞品对比分析、新产品立项评估、产品复盘与优化时

## 前置检查

### ⚠️ 飞书/Feishu 移动端交付说明

当在飞书移动端交付分析报告时：
- **不要使用深层嵌套的 Markdown 表格** — 飞书移动端渲染大表格时阅读体验极差，用户明确反馈过
- **改为生成移动优先的 HTML 报告**，保存后通过飞书发送文件（`MEDIA:/path` 语法）
- HTML 风格：卡片式布局、渐变 Header、圆角白色容器、色块标注（红/绿/琥珀）、P0/P1/P2 优先级标签
- 参考模板：[templates/mobile-report-feishu.html](templates/mobile-report-feishu.html) — 使用 `{{占位符}}` 结构填充内容
- 如需在飞书消息中文字展示核心结论，用简短 bullet list（≤3条）+ 一句话总结收尾，而不是完整报告文本

### 步骤 1：信息收集与背景梳理
- 明确分析对象：产品名称、版本/阶段、行业领域
- 确定分析目的：战略规划/竞品分析/立项评估/产品复盘
- 收集基础信息：
  - 产品官网、应用商店、官方介绍
  - 目标用户反馈、评价、社交媒体声音
  - 行业报告、市场数据、竞品信息
  - 产品功能列表、定价策略

### 步骤 2：五维深度分析
按以下五个维度进行系统化分析，每个维度参考 [references/dimensions.md](references/dimensions.md) 的详细要点：

#### 2.1 战略层面
- 市场定位：在行业生态中的位置与差异化
- 产品愿景：长期目标和价值主张
- 资源投入：技术、人力、资金的投入重点
- 发展路径：短期、中期、长期演进规划

#### 2.2 目标客群
- 用户画像：人口特征、行为特征、心理特征
- 需求层次：核心需求、次要需求、潜在需求
- 使用场景：高频场景、低频场景、边缘场景
- 决策因素：购买动机、使用动机、留存因素

#### 2.3 商业模式
- 价值主张：为用户解决的核心问题
- 盈利模式：收入来源、收费方式、定价策略
- 成本结构：获客成本、运营成本、研发成本
- 增长策略：拉新、激活、留存、变现、传播

#### 2.4 关键产品竞争力
- 技术壁垒：核心技术、专利、技术领先性
- 产品优势：功能创新、性能优势、体验差异
- 生态优势：合作伙伴、第三方集成、平台化程度
- 网络效应：用户增长如何提升产品价值

#### 2.5 体验评估
- 用户旅程：从接触到使用的全流程体验
- 核心体验点：关键场景的交互体验、视觉体验、性能体验
- 痛点与爽点：用户吐槽点、用户喜爱点
- 满意度指标：NPS、满意度评分、留存率

### 步骤 3：经典框架辅助分析（可选）
需要深度分析时，参考 [references/frameworks.md](references/frameworks.md) 中的经典框架：
- SWOT 分析：优势、劣势、机会、威胁
- PESTEL 分析：宏观环境因素
- 商业模式画布：九大模块结构化分析
- 价值主张画布：用户价值与产品价值匹配
- 竞争分析矩阵：多维度竞品对比
- 用户体验地图：用户情绪曲线与触点分析

### 步骤 4：整合洞察与战略输出
综合五个维度的分析结果，输出结构化洞察报告：

#### 4.1 核心洞察提炼
- 识别产品最关键的 3-5 个优势
- 识别最关键的 3-5 个风险与挑战
- 总结核心差异化定位

#### 4.2 战略建议
- 短期行动建议（3-6 个月）
- 中期战略方向（6-18 个月）
- 长期发展愿景（18 个月以上）
- 资源投入优先级建议

#### 4.3 风险与机会
- 主要风险点与应对策略
- 关键机会点与抓取路径

#### 4.4 输出报告
根据分析深度选择输出形式：
- **简版**：关键洞察 + 3 条战略建议（1-2 页）
- **标准版**：五维分析 + 框架辅助 + 战略建议（5-8 页）
- **深度版**：完整五维分析 + 多框架交叉 + 详细行动计划（15 页以上）

参考 [assets/template.md](assets/template.md) 的报告模板结构进行输出。

## 注意事项

### 🔍 来源可追溯性验证（用户要求，适用于所有分析摘要类任务）

**当从已有源材料（报告/文章/文档）中提炼总结或生成洞察摘要时，必须在输出前执行以下验证：**

1. **逐句溯源** — 写完后，对摘要/洞察中的每一个主张，逐一检查是否能在源文档中找到原文支撑
2. **分类标注** — 每个主张分为三类：
   - ✅ **直接支撑**：源文档中有原文直接对应
   - ⚠️ **合理推论**：源文档未直接表述但可通过逻辑推导得出，需在输出中标注「推论自：xxx」
   - ❌ **无来源**：源文档不支持的主张，必须删除或补充来源后再输出
3. **行号标记** — 在交付件中标注每条结论对应的源文档行号或章节号，便于用户验证
4. **交叉验证** — 同一数据点尽量有多个独立来源印证（如 Gartner + IDC 同时指向同一趋势）
5. **推论诚实** — 当必须做推论时（如将行业趋势映射到特定用户群体），明确说明「这是基于xxx数据的推断」
6. **典型陷阱**（据此对话经验）：
   - ❌ 将源文档中的「手机/移动端」泛化为「PC」— 原报告若只覆盖手机，不得引入 PC
   - ❌ 在无成本数据的报告中添加「成本」相关定性描述（如"一个人的成本"）
   - ❌ 将源文档的三层架构（云/边/端）自行细化为具体设备角色分工，除非源文档明确指定
   - ❌ 引入源文档不存在的术语（如"Agent 即服务"在源报告中未出现，需标注为推论）

> **此验证步骤不可跳过。** 用户会明确要求「对照源文档仔细审视」，发现逻辑缺口后会要求纠正。与其等用户指出，不如在交付前自行完成。

#### 响应数据质疑：当用户对某个数据点提出疑问

用户有时会对报告中的具体数字提出质疑（如「22亿是2.2亿还是22亿？」）。这是正常的严谨性检查，按以下流程响应：

1. **溯源到原始出处** — 找到报告中该数据对应的原始来源（如 IDC 官方链接、Gartner 报告页），不要只依赖二手引用
2. **多源交叉验证** — 用 `web_search` 搜索至少 2-3 个独立来源的同一数据，确保一致性。不同媒体的报道可能有出入，以官方机构原文为准
3. **数学自洽检查** — 验证数据内部逻辑是否自洽。例如 2025年 2,860万 × 80x ≈ 22.88亿，与 22.16亿 在同一量级，数学上合理
4. **中英文计数转换** — 注意中文「亿」与英文数字的对应关系：
   - 1 亿 = 100 million
   - 10 亿 = 1 billion
   - 22 亿 = 2.2 billion = 2,200 million
   - 容易混淆点：22亿 ≠ 2.2亿（后者是 220 million）
5. **给出明确结论** — 「数据准确」或「数据有误，需要修正」，不要模糊表述
6. **如果数据有误** — 立即修正报告，并在报告中标注修正后的来源

- 聚焦核心：避免面面俱到，突出最关键的洞察点
- 数据支撑：尽量引用具体数据、用户反馈、市场信息佐证
- 结构化表达：使用清晰的层级结构、列表、表格提升可读性
- 可落地性：战略建议必须具体、可执行、有优先级
- 仅在需要深度分析时读取经典框架，保持上下文简洁
- **飞书移动端输出**：交付在飞书...
- **格式优先级**：HTML 移动端报告 > 简短文字摘要 > 复杂 Markdown 表格（飞书移动端场景下）
- ⚡ **Web UI 调用禁止修改本 skill**：通过 portal/webhook 调用时，agent 不得对 skill 进行任何修改/优化/更新。Skill 是固定版本，防止被污染。

### ⚠️ Portal 调用约束
当通过运营门户（8899）的洞察模块提交 task 时（即 WebUI 调用而非 chat 对话），**禁止对 skill 本身做任何修改/优化/更新**。原因：门户调用的 skill 是固定版本，防止多人并发使用时 skill 被污染。portal 前端已展示当前调用的 skill 名称（`product-insight` / `ux-evaluator` / `ux-demo`）。

### ⚠️ UI 表格布局原则（用户明确反馈过）
在 HTML 页面中渲染表格时（如洞察/任务历史），遵循以下布局规则以避免内容换行、列宽错乱：
1. **`table-layout:fixed`** — 固定布局模式，列宽由 `<th>` 设定决定，防止内容撑开
2. **非数据列必须 `white-space:nowrap`** — 类型、状态、操作、ID、时间等列加 `nowrap`，只有标题/描述列允许自动换行或截断
3. **最小列宽参考** — 类型 ≥110px（emoji+中文）、状态 ≥110px（带点徽章）、操作 ≥100px（按钮+icon）、时间 ≥160px（两行时间戳）、ID ≥80px
4. **小屏处理** — 外层容器设 `overflow-x:auto` 保证表格可横向滑动，减少内边距但保留列宽
5. **状态/操作组件** — `.badge`、`.btn` 等内联组件追加 `white-space:nowrap`，禁止换行

## 扩展用法：行业趋势研究（2026-2030 宏观洞察）

当需要大规模行业趋势研究（如「2026-2030行业趋势」类任务）时，使用以下工作流替代逐一手动搜索：

### 工作流：并行研究 → 整合分析 → HTML报告

#### 阶段一：并行数据采集（delegate_task 批量模式）
将研究任务拆分为 3 个独立方向，用 `delegate_task` 的 `tasks` 数组并行执行（如该工具可用）：

| 任务 | 研究方向 | 工具集 |
|------|----------|--------|
| 1 | 宏观趋势：Gartner/McKinsey/IDC/Deloitte 报告数据 | `web` |
| 2 | Agent化趋势：AI Agent市场数据、技术栈、框架演进 | `web` |
| 3 | 移动化趋势：端侧AI、边缘计算、云厂商Agent战略 | `web` |

每个子任务要求：
- 用 `web_search` 搜索中英文资料
- 返回结构化数据 + 来源URL
- 用中文回答（最终报告为中文）

**⚠️ delegate_task 不可用时的回退方案：**
如果 `delegate_task` 工具不可用（如 Cockpit Worker 运行环境中），改用**顺序 `web_search` 调用**替代并行采集。按 宏观 → Agent → 移动化 顺序依次搜索，每个方向搜索 2-3 个不同关键词以覆盖多源数据。注意：顺序搜索效率低，分配给每个方向的调用次数要克制（≤3次/keyword），避免总调用数过大。数据交叉验证（同一数据至少2个独立来源）仍然必须坚持。

#### 阶段二：整合分析
收到子任务结果后，按以下维度整合：
1. **市场验证** — 每个趋势的市场规模、CAGR、权威预测数据
2. **交叉验证** — 同一数据是否有多个权威来源印证
3. **趋势判断** — 明确标注 ✅支撑 / ❌不支撑 / ⚠️需观察
4. **交汇分析** — 移动化×Agent化的交汇点（技术、市场、竞争）
5. **创业启示** — 针对 OPC/创业者的具体建议（P0/P1/P2 优先级）

#### 阶段三：生成移动端HTML报告（含增长趋势可视化）
在生成报告前，先检查 `~/.hermes/reports/` 下是否有与 topic 关键词匹配的近期报告（最近7天内）：
- 如果存在且内容已满足任务要求（已有2026-2030增长趋势图、移动端适配等），**优先复用**并更新其生成时间戳
- 如果不存在或不满足要求，再按以下模板生成新报告

**⚠️ 增长趋势可视化要求（2026-2030 逐年数据）：**

报告中的增长趋势图表（柱状图/曲线图）必须满足以下要求：

1. **逐年数据点** — 必须包含 2026、2027、2028、2029、2030 共5个年度的具体数值，不可缺少年份
2. **增长率标注** — 每个数据点上方/旁边必须标注同比增速（如：+18.5%），格式为百分比保留一位小数
3. **柱状图必选** — 市场规模走势必须使用柱状图展示，柱子颜色使用渐变色 #00d4aa → #0099ff
4. **曲线图可选** — 可在柱状图基础上叠加增长率折线（右侧Y轴），使用 #f5a623 颜色
5. **数据标签** — 每根柱子/数据点上方必须直接显示具体数值（如：$850B），**禁止仅靠鼠标悬停显示**。增长率在数据值下方用小字号显示
6. **来源标注** — 图表底部必须标注数据来源机构名称（如：数据来源：Gartner, 2026年1月），使用彩色 badge 区分不同来源
7. **CAGR说明** — 图表旁或下方用卡片展示 2026-2030 年复合年增长率（CAGR），格式：`XX% CAGR (2026-2030)`
8. **渲染方式选择** — 根据交付目标选择：
   - **网页部署（有JS运行时）**：使用 Chart.js v4 + `chartjs-plugin-datalabels@2.2.0` 实现静态数据标签，利用插件自动在每个数据点上方显示数值，无需鼠标悬停
   - **飞书/离线附件（无JS）**：使用纯 HTML + CSS 实现（CSS flexbox 柱状图 + CSS 伪元素标注数值，见下方CSS实现）
9. **Roadmap/路标时间线** — 当报告包含技术演进路径时（如「2025-2030 AI Agent技术演化路标」），使用垂直时间线布局，每个时间节点包含：年份徽标 + 阶段标题 + 详细说明 + 技术标签（model/infra/app/eco/device），颜色编码区分不同类别

### UI 布局细节（用户对体验的严格要求）

当在门户系统中交付 HTML 页面（如洞察库表格）时，以下细节必须精确处理。用户发现不当时会明确指出：

| 问题 | 用户反馈 | 正确做法 |
|------|---------|---------|
| 列宽不足导致换行 | "行业洞察等还是换行了" | 设置固定列宽（px）+ `white-space:nowrap`，而非靠浏览器 auto 撑开 |
| 列宽不受控 | 内容挤占相邻列 | 表格添加 `table-layout:fixed`，每列显式声明 `width` |
| 徽章/按钮换行 | "状态、操作等也都不正常" | 对 `span.badge`、`button` 等组件添加 `white-space:nowrap` |
| 小屏截断 | 内容被裁切 | 容器设 `overflow-x:auto` 保证可横向滚动 |
| 提示信息被压缩 | 右上方提示框文字看不清 | `max-width` 给足余量（至少 260-280px） |

**通用原则：** 表格所有非数据列（类型、ID、状态、操作、时间）必须 `white-space:nowrap`，只有标题/描述列允许换行或截断。手机端减少内边距但保留水平滚动。

### 示例 1：新产品立项评估
**场景**：团队计划开发一款面向中小企业的 CRM 系统，需要评估市场机会与产品定位
**执行方式**：
- 收集：中小企业 CRM 市场报告、竞品（Salesforce 简易版、钉钉 CRM 等）信息
- 分析：重点分析目标客群（中小企业特征、需求层次）、商业模式（收费模式、获客成本）、关键竞争力（差异化点）
- 输出：简版报告，聚焦市场机会与差异化定位

### 示例 2：竞品深度分析
**场景**：分析竞品 Notion 的核心竞争力，为自家产品优化提供方向
**执行方式**：
- 收集：Notion 官网、用户评价、功能列表、定价策略
- 分析：五维全面分析，重点在关键竞争力（技术壁垒、生态优势）、体验评估（用户旅程、核心体验点）
- 框架辅助：使用竞争分析矩阵、用户体验地图
- 输出：标准版报告，包含竞品优势分析与本产品优化建议

### 示例 4：云厂商/平台型竞品分层分析
**场景**：深入分析一个多层平台型产品（如阿里云 OPC、华为云 AgentArts、腾讯云五层Matrix等），需要跨层对比竞品
**执行方式**：
- 收集：各厂商产品官网、定价页、发布会资料、开发者社区文章
- 分析：分层拆解（芯片→模型→平台→Agent→应用），每层用 P0/P1/P2 评级
- 框架辅助：分层对比矩阵 + SWOT + `references/competitor-architecture-pattern.md`
- 输出：深度版 HTML 报告，含逐层产品卡、竞争对手对比表、P0/P1/P2 战略建议

### 示例 3：产品战略规划
**场景**：一款上线 1 年的 SaaS 产品，需要制定未来 18 个月的发展战略
**执行方式**：
- 收集：产品现状数据、用户反馈、市场趋势
- 分析：五维全面分析，重点在战略层面（发展路径）、商业模式（增长策略）、关键竞争力（技术演进）
- 框架辅助：使用 SWOT 分析、商业模式画布
- 输出：深度版报告，包含短中长期战略规划与资源投入建议
