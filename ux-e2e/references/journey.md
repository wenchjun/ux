# 客户旅程框架

> 执行步骤 3 时参考。SMB 客户在公有云场景下的典型端到端旅程与关键触点。

---

## 一、旅程四阶段总览

```
感知 (Awareness)  →  购买 (Purchase)  →  使用 (Usage)  →  反馈 (Feedback)
─────────────────────────────────────────────────────────────────────────
产品详情页          套餐/购买页          控制台/操作页       客服/评价/社区
价值认知            决策与付费           任务完成             问题解决
"这是什么？"        "值不值？"          "怎么用？"           "有问题找谁？"
```

---

## 二、阶段一：感知（Awareness）

### 用户目标
- 快速了解产品是什么、能解决什么问题
- 判断产品是否值得进一步了解
- 建立初步信任（品牌、安全性、可靠性）

### 关键触点
| 触点 | 用户行为 | 体验指标 |
|------|----------|----------|
| 搜索引擎结果 | 搜索问题 → 点击链接 | 标题/描述是否匹配用户意图 |
| 落地页首屏 | 5 秒内扫描标题和主视觉 | 能否一句话说清产品价值 |
| 功能展示区 | 浏览功能列表、截图 | 功能描述是否符合用户场景 |
| 定价概览 | 寻找价格信息 | 价格是否透明、容易找到 |
| 客户案例 | 查看谁在用 | 是否有同行业/同规模客户 |
| 注册/试用入口 | 决定是否尝试 | CTA是否醒目、门槛是否低 |

### SMB 客户特征
- 通常由创始人/技术负责人直接搜索并决策
- 5 秒内无法理解产品价值就会跳出
- 对"免费试用"和"透明定价"反应积极
- 重视同行案例（"和我差不多的公司在用吗？"）

### 评估要点
1. Hero Banner 能否 5 秒传达价值
2. 核心功能是否首屏可见
3. 是否提供了社会证明（客户logo、案例、评价）
4. 从落地页到购买页的路径是否顺畅
5. 移动端浏览体验是否合格

---

## 三、阶段二：购买（Purchase）

### 用户目标
- 理解各套餐差异，找到适合自己的方案
- 确认价格透明、无隐藏费用
- 完成支付，流程顺畅

### 关键触点
| 触点 | 用户行为 | 体验指标 |
|------|----------|----------|
| 套餐对比 | 横向比较各套餐 | 差异是否一目了然 |
| 价格计算 | 确认最终价格 | 价格是否含税、是否有额外费用 |
| 注册/登录 | 创建账号或登录 | 是否强制提前注册 |
| 支付流程 | 填写信息、完成支付 | 步骤数量、支付方式、安全性 |
| 购买确认 | 收到购买成功通知 | 确认信息是否清晰完整 |

### SMB 客户特征
- 价格敏感，会仔细比较各套餐性价比
- 倾向自助购买，不愿联系销售
- 犹豫期短但容易放弃（任何摩擦都可能导致流失）
- 需要"可退订"的信心保障

### 评估要点
1. 套餐数量是否在 3-5 个（希克定律）
2. 是否有一个明确的"推荐"或"最受欢迎"选项
3. 价格信息是否完整透明（含税、计费周期、超额政策）
4. CTA 按钮是否醒目、文案是否明确
5. 购买流程步骤数（建议 ≤ 3 步）
6. 移动端能否完成购买
7. 是否有退款/退订政策的明确说明
8. 支付方式是否满足目标市场（支付宝/微信）

---

## 四、阶段三：使用（Usage）

### 用户目标
- 快速上手，完成首次任务
- 高效完成日常操作
- 遇到问题时能自助解决

### 关键触点
| 触点 | 用户行为 | 体验指标 |
|------|----------|----------|
| 新手引导 | 首次进入产品 | 是否有引导、示例数据 |
| 核心任务 | 完成主要操作（如创建资源） | 操作步骤数、默认值智能度 |
| 导航 | 在不同功能间切换 | 菜单结构是否清晰 |
| 搜索 | 搜索功能/资源 | 搜索结果是否准确 |
| 错误处理 | 遇到操作失败 | 错误信息是否可理解、可解决 |
| 空状态 | 新用户看到空白页面 | 是否有引导性内容 |

### SMB 客户特征
- IT 能力有限，需要低门槛上手
- 可能没有专职运维，偏好简单直观的操作
- 时间紧张，希望快速完成任务
- 容易因困惑而流失

### 评估要点
1. 是否有新手引导或快速开始向导
2. 核心任务的操作步骤数
3. 默认值/预配置是否减少用户输入
4. 导航是否符合用户心智模型
5. 操作反馈是否即时、明确
6. 错误信息是否友好、可操作
7. 空状态是否有引导（而非空白页）
8. 高频操作是否有快捷入口

---

## 五、阶段四：反馈（Feedback）

### 用户目标
- 遇到问题时能找到帮助
- 提交反馈/建议能被响应
- 持续获得产品更新和价值感知

### 关键触点
| 触点 | 用户行为 | 体验指标 |
|------|----------|----------|
| 帮助文档 | 搜索使用教程 | 文档是否齐全、可搜索 |
| 客服入口 | 联系客服 | 入口是否容易找到 |
| 问题反馈 | 提交 bug 或建议 | 提交流程是否简便 |
| 评价渠道 | 查看/留下评价 | 是否有公开评价入口 |
| 续费提醒 | 收到续费通知 | 提醒时机是否合理 |
| 升级引导 | 发现更高套餐 | 升级路径是否清晰 |

### SMB 客户特征
- 偏好自助解决（文档、FAQ）
- 期待即时响应（聊天/工单）
- 需要清晰的升级路径（业务增长时）

### 评估要点
1. 客服入口是否容易找到（至少 2 个位置）
2. 帮助文档是否有搜索功能
3. FAQ 是否覆盖核心问题
4. 是否有用户社区/论坛
5. 错误信息中是否包含求助链接
6. 是否有定期回访/满意度调研

---

## 六、体验一致性检查（跨阶段）

| 检查项 | 说明 |
|--------|------|
| 视觉一致性 | 详情页 → 购买页 → 控制台的颜色/字体/组件是否统一 |
| 术语一致性 | "套餐""计划""方案"在不同页面是否统一 |
| 交互一致性 | 按钮样式、导航模式、表单行为是否跨页面一致 |
| 承诺一致性 | 详情页承诺的功能在控制台是否真的可用 |
| 价格一致性 | 详情页和购买页的价格/规格是否一致 |

---

## 七、转化漏斗分析框架

```
                   访问落地页  100%
                       ↓
                   理解产品价值  ~60%  ← 流失：价值主张不清
                       ↓
                   查看套餐价格  ~35%  ← 流失：找不到价格/价格吓退
                       ↓
                   点击 CTA      ~20%  ← 流失：按钮不醒目/文案模糊
                       ↓
                   开始注册/购买  ~12%  ← 流失：注册摩擦/流程过长
                       ↓
                   完成支付      ~5%   ← 流失：支付问题/信心不足
```

在每个阶段标注实际发现的流失原因和改进建议。
