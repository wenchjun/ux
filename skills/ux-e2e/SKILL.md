# 端到端网页体验测评

## 任务目标
- 本 Skill 用于：基于用户提供的关键网页（产品详情页、购买页、操作页），生成一份端到端的用户体验测评报告
- 适用领域：公有云产品，主要面向 SMB（中小企业）客户
- 核心输出：HTML 格式的专业测评报告，覆盖感知→购买→使用→反馈完整客户旅程
- 触发条件：用户要求进行网页体验测评、UX评测、产品体验分析、端到端体验评估

## 前置条件
- 用户需提供 2-4 个关键网页 URL，至少包含：
  - **产品详情页**（感知环节）：产品介绍、价值主张、功能展示
  - **产品购买页**（购买环节）：套餐展示、定价、购买流程
  - **产品操作页/控制台**（使用环节）：实际操作界面（如有）
- 如果用户只提供了一个页面，先基于该页面完成感知和购买环节的测评，并询问是否还有其他页面

## 测评框架

### 客户旅程四阶段

| 阶段 | 核心页面 | 关键目标 | 测评重点 |
|------|----------|----------|----------|
| **感知** Awareness | 产品详情页/落地页 | 让用户快速理解产品价值并产生兴趣 | 信息架构、价值传达、视觉吸引 |
| **购买** Purchase | 套餐页/购买页 | 降低决策成本，促进转化 | 定价清晰度、CTA设计、信任构建 |
| **使用** Usage | 控制台/操作页 | 让用户高效完成任务 | 导航效率、交互流畅度、容错性 |
| **反馈** Feedback | 客服入口/评价渠道 | 收集用户声音，形成闭环 | 反馈路径、问题解决、持续触达 |

### 五层评价维度

#### 1. 视觉层（Surface）
- 布局与网格系统：信息层级是否清晰、视觉重心是否明确
- 色彩与对比度：品牌色一致性、文字可读性（WCAG 对比度标准）
- 排版与字体：字体层级、行间距、可读性
- 图标与插画：风格一致性、语义清晰度
- 响应式适配：移动端/平板/PC 的布局适配

#### 2. 交互层（Interaction）
- **费茨定律（Fitts' Law）**：核心操作按钮的尺寸与位置是否易于点击；移动端触控区域是否 ≥ 44×44px
- **希克定律（Hick's Law）**：选择数量是否合理——每增加一个选项，决策时间对数级增长
- **格式塔原则**：接近性、相似性、连续性——相关元素是否视觉分组
- 导航效率：用户能否在 3 次点击内到达目标
- 系统反馈：操作后是否有即时、明确的反馈（加载态、成功态、错误态）
- 容错与恢复：是否支持撤销、是否有确认弹窗、错误信息是否友好

#### 3. 文案层（Copy）
- 价值主张：核心标题能否在 5 秒内传达产品价值
- 行动引导（CTA）：按钮文案是否明确可预期（"立即订阅" vs "提交"）
- 信息清晰度：行业术语是否被解释、价格是否透明（含税/不含税）
- 一致性：同一概念在不同页面是否使用统一术语
- 本地化质量：中文表达是否自然、是否有翻译腔

#### 4. 信息架构层（Structure）
- 内容组织：信息是否按用户心智模型分类
- 导航结构：面包屑、侧边栏、标签页是否清晰
- 搜索与筛选：查找效率、筛选维度是否合理

#### 5. 战略层（Strategy）
- 用户目标匹配：页面是否解决用户到访的核心任务
- 商业目标平衡：产品目标与用户需求是否有冲突（如强制注册时机）
- SMB 适配度：是否考虑中小企业客户的决策特征（决策链短、价格敏感、IT能力有限）

## 操作步骤

### 步骤 1：页面采集与分析

对用户提供的每个 URL 执行以下操作：

1. 使用 WebFetch 工具获取页面内容
2. 若 WebFetch 被拦截，使用 Bash(curl) 抓取 HTML 源码
3. 从 HTML 中提取关键信息：
   - 页面标题、meta description
   - 所有标题层级（h1-h6）
   - CTA 按钮文案与链接
   - 定价信息（产品名、价格、规格）
   - 表单元素（数量、类型、必填项）
   - 导航结构
   - 图片 alt 属性
   - 语义化标签使用（header/nav/main/section/footer）
4. 记录技术指标：
   - 外部资源数量（JS/CSS）
   - HTML 文件大小
   - viewport 配置
   - CSP 安全策略
   - 是否使用 noscript 降级

### 步骤 2：分页面逐项评估

对每个页面，按照五层评价维度逐项评分（1-5分），记录具体问题与截图位置的文字描述。

**评分标准**：
- 5分：行业标杆，体验优秀
- 4分：良好，有小幅优化空间
- 3分：及格，存在明显问题
- 2分：较差，影响用户完成任务
- 1分：严重缺陷，阻断用户体验

评分细则参考 [references/scoring.md](references/scoring.md)。

### 步骤 3：端到端客户旅程串联

将各页面的评估结果串联为完整客户旅程：

1. 绘制用户从"首次接触→购买决策→日常使用→问题反馈"的完整路径
2. 识别各阶段之间的**断层点**（如：详情页承诺的功能在控制台找不到）
3. 计算**转化漏斗流失点**（如：移动端无法购买、未登录跳转登录页）
4. 评估**体验一致性**：视觉风格、交互模式、术语是否跨页面一致

### 步骤 4：关键原则诊断

基于交互设计经典原则进行专项诊断，参考 [references/principles.md](references/principles.md)：

- **费茨定律**：关键按钮的尺寸、位置、点击区域
- **希克定律**：套餐数量、导航选项是否过多
- **特斯拉定律（Tesler's Law）**：复杂度是否被合理转移（让系统承担而非用户）
- **接近定律**：相关功能是否靠近、不相关的是否分离
- **防错原则**：是否有确认机制、危险操作是否有二次确认
- **识别而非回忆**：选项是否可见、用户是否需要记住信息

### 步骤 5：输出 HTML 测评报告

使用 [assets/report-template.html](assets/report-template.html) 作为基础模板，生成最终的 HTML 测评报告。

报告必须包含以下结构：
1. **报告头部**：产品名称、测评日期、测评页面列表、综合评分
2. **总览仪表盘**：六维度雷达/星级评分 + 关键发现摘要
3. **客户旅程总览**：感知→购买→使用→反馈的体验曲线
4. **分页面详细评估**：每个页面的五层维度详细评分与问题清单
5. **交互设计原则诊断**：费茨定律、希克定律等关键原则的合规情况
6. **转化漏斗分析**：各阶段流失点与改进建议
7. **问题清单（优先级排序）**：P0阻断/P1重要/P2优化
8. **改进建议路线图**：短期/中期/长期行动计划
9. **竞品对比参考**（如有）

## 注意事项

### 测评原则
- **以用户为中心**：始终从 SMB 客户（而非产品经理或开发者）的视角评估体验
- **数据驱动**：每个问题必须附带具体证据（页面位置、HTML 代码片段、截图描述）
- **可落地**：每条建议必须具体、可执行，包含"当前→建议"的对比
- **优先级明确**：使用 P0/P1/P2 标记问题严重程度，帮助团队聚焦关键问题

### SMB 客户特征
- 决策链短：通常 1-2 人即可决策，购买决策快但犹豫期短
- 价格敏感：对中小企业的预算有限，需要清晰的 ROI 感知
- IT 能力有限：可能没有专职运维，产品需要低门槛、易上手
- 看重口碑：容易受同行评价和案例影响

### 输出规范
- 报告必须是完整的、可独立打开的 HTML 文件（内嵌 CSS）
- 响应式设计，适配移动端查看
- 使用专业的视觉风格，避免花哨，注重信息层级
- 所有评分使用星级（⭐）展示，一目了然
- 关键问题使用红色/黄色/绿色标签区分严重程度

## 使用示例

### 示例 1：完整三页面测评
**场景**：用户提供产品详情页、购买页、控制台页，做完整端到端测评
**输入**：
- 产品详情页：https://example.com/product/detail
- 购买页：https://example.com/pricing
- 控制台：https://console.example.com/dashboard
**执行方式**：
- 逐个采集三个页面的内容
- 每个页面按五层维度评分
- 串联客户旅程，标记断层点
- 按经典原则诊断关键交互
- 输出完整 HTML 报告

### 示例 2：单页面快速测评
**场景**：用户只提供落地页，快速评估感知环节
**输入**：https://example.com/landing
**执行方式**：
- 采集页面内容并评估
- 聚焦感知阶段的视觉/文案/信息架构
- 标注页面中的购买入口和转化路径
- 生成聚焦"感知+初步购买意向"的精简报告
- 提示用户补充其他页面可做更完整的端到端测评

### 示例 3：竞品体验对比
**场景**：用户提供两个竞品的产品页面，对比体验差异
**输入**：
- 产品A详情页 + 购买页
- 产品B详情页 + 购买页
**执行方式**：
- 分别评估两个产品
- 在各维度中标注 A vs B 的差异
- 报告增加竞品对比矩阵
- 输出包含"可借鉴亮点"与"差异化机会"
